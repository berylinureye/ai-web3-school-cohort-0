### 1. Dev Stack 最小实践 6 步

Handbook 的最小实践：

1. 用 Remix 部署一个极简计数器合约，包含 `count()`、`increment()` 和 `CountChanged` event。
2. 用 Hardhat 或 Foundry 建一个本地工程，并写测试覆盖初始值和 `increment()` 后的状态变化。
3. 记录合约地址、ABI、部署网络、部署账号和交易哈希。
4. 用 viem 或 wagmi 从前端读取 `count()`，再发起一次 `increment()` 交易。
5. 在区块浏览器或本地日志里确认 event 被正确发出。
6. 写下这条链路中哪些信息必须进版本控制，哪些必须放在 `.env` 或密钥管理里。

理解版拆解：

```text
合约逻辑
  ↓
工程化测试
  ↓
部署记录
  ↓
前端接链
  ↓
链上验证
  ↓
安全边界
```

今天先做理解版，不急着完整搭 Hardhat / Foundry 工程。对我当前阶段，更重要的是把这条链路转成 Week 1 流程图和 Week 2 Wallet / Permission 方向的材料。

### 2. 小考纠偏：Counter / 网络地址 / Git 边界

我的回答：

- `count` 是用来增加次数；`increment` 是让次数增加；`CountChanged event` 是 check 变化。
- 部署网络说明合约部署在哪，合约地址说明地址在哪；前端拿错网络或地址会报错。
- tx hash / 合约地址可以写进 GitHub，私钥 / API key 不行。

纠偏后：

- `count()` 是**读当前次数**，不改链，不花 gas。
- `increment()` 才是**让次数增加**，会改链，需要交易和 gas。
- `CountChanged event` 不是“检查动作”本身，而是 `increment()` 成功改变状态后发出的链上日志。前端、Blockscout、索引器可以通过 event 确认变化发生过。
- 部署网络必须记录，因为 Sepolia / Base / Mainnet 是不同链；同一个地址在不同网络可能不是同一个合约。
- 合约地址必须记录，因为前端要知道自己正在跟哪个合约账户交互。拿错地址轻则报错，重则和错误合约交互。
- 可以进 GitHub：ABI、测试网合约地址、tx hash、合约源码、测试代码、学习笔记。
- 不能进 GitHub：私钥、助记词、API key、RPC secret、任何能代表我签名或付费的 token。

复考：

> 如果前端页面显示的是 Sepolia 的合约地址，但用户钱包当前连在 Mainnet，最合理的 UI 应该怎么做？

我的回答：提示用户切换到 Sepolia。

确认：正确。前端 / Agent 不能让用户在错误网络上继续点交易，更不能自动帮用户签名。正确流程是先识别网络不匹配，再提示用户切换网络，切换完成后再允许继续操作。

### 4. Agent Wallet：不是给 AI 一把私钥

Handbook 核心句：

> Agent Wallet 不是“给 AI 一个钱包私钥”这么简单。真正要解决的问题是：Agent 可以代表用户做哪些链上动作，这些动作有没有额度、时间、对象和风险限制，用户什么时候必须重新确认。

第一性原理：

> 控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。

我的初始回答：

- Agent Wallet 需要有思考和决策能力；单纯给 AI 私钥没有决策链路。
- 概率系统过于简单、没有兜底、没有决策。
- 最小权限可以限制最大金额、approve 次数、可执行范围。

纠偏后：

- “不是给 AI 一个私钥”的核心不是 AI 缺少决策链路，而是**私钥给的是完整控制权**。一旦 Agent 拿到 EOA 私钥，它可以转账、approve、调用任何合约，用户很难限制它“只能做某些低风险动作”。
- “控制权不能交给概率系统”的意思不是 AI 太简单或没有决策，而是 AI 的判断来自模型、上下文和工具返回值，可能被 prompt injection、错误网页、不完整状态或幻觉带偏。它可以很有用，但不应该拥有不可限制的资产控制权。
- 更合理的 Agent Wallet 是：Agent 有一定行动空间，但这个空间必须可检查、可限制、可撤销。

最小权限边界可以包括：

| 限制 | 例子 |
|---|---|
| 金额限制 | 每日最多 5 USDC，单笔最多 1 USDC |
| 对象限制 | 只能调用白名单合约 |
| 动作限制 | 默认只读；写操作只允许少数函数；approve 必须人工确认 |
| 时间限制 | session key 24 小时后过期 |
| 网络限制 | 默认只能 Sepolia，主网必须人工确认 |
| 撤销能力 | 用户可随时 revoke / pause |

复考：

> 如果 Agent 拿到一个 session key，只允许它在 24 小时内、Sepolia 上、调用 Counter 合约的 `increment()`，单笔不能转 ETH。这比直接给它 EOA 私钥安全在哪里？

我的回答：

- Sepolia 是测试网络，更安全。
- 限制不能转 ETH，能降低损失。
- 直接给 EOA 私钥等于完整使用权限，风险太大。

确认：正确。补充三点：

- session key 有时间限制，24 小时后自动失效。
- 它只能调用指定合约 / 指定函数，不能任意调用未知合约。
- 它可以被 revoke / pause，而 EOA 私钥泄漏后通常只能紧急迁移资产。

### 5. Smart Account / Safe / Session Key 小考

Handbook 这一段的核心：

- **Smart Account**：带规则的钱包账户。
- **Safe**：适合团队、DAO 和高价值账户的多签 / 智能账户基础设施。
- **Session Key**：临时、有限权限的小钥匙，本质是一组受限能力。

我的回答：

1. Smart Account 的权限比普通 EOA 私钥小；最大区别是使用 session key，权限有限，用户可以随时 revoke。
2. Safe 解决多人 / 团队确认，不让一个人或一个 Agent 单独控制重要资产。
3. Session key 可以让 Agent 调用限定权限，比如只在 Sepolia 或调用特定函数，但不能 approve，用户可以 revoke。

纠偏：

- Smart Account 不是“权限一定更小”，而是**账户规则可编程**。它可以表达限额、白名单、session key、恢复、多签、暂停等规则。session key 是 Smart Account 可以支持的一种能力，不是 Smart Account 的全部。
- Safe 的理解正确：它把高价值操作放进多人确认流程，让 Agent 可以参与流程但不能绕过治理。
- Session key 的理解正确：它不是小号私钥，而是临时、可撤销、范围受限的能力。

### 6. Counter 合约为什么没业务意义但仍值得学

今天问到：Counter 合约感觉没有意义，还有别的合约吗？

结论：

> Counter 本身几乎没有业务价值，它是链上 Hello World。它的价值是展示智能合约最小骨架：读状态、写状态、发 event、链上验证。

对照表：

| Counter | ERC-20 | Safe | Paywall |
|---|---|---|---|
| `count()` | `balanceOf()` | `getOwners()` | `hasPaid()` |
| `increment()` | `transfer()` | `execTransaction()` | `pay()` |
| `CountChanged` | `Transfer` | `ExecutionSuccess` | `PaymentReceived` |

我的回答：Counter 没业务价值但适合学 dev-stack，因为它包含的部分和背后的逻辑是一样的。

确认：正确。复杂合约的业务不同，但底层都离不开读状态、写状态、权限检查、event 和链上验证。

### 7. Policy / Guard / Simulation / Revocation

Handbook 这一段的核心：

- **Policy**：把“Agent 可以做什么”写成系统能检查的规则。
- **Guard**：确定性拦截层，负责拒绝不符合 policy 的交易或工具调用。
- **Simulation**：在签名和广播前预演交易结果。
- **Revocation**：权限如何被用户或系统及时收回。

关于 Guard 的关键纠偏：

> Guard 不能保证“不发生问题”，它只能保证“不违反已经写清楚、机器能检查的 policy”。

我的理解：

> Guard 是用来保证规定的 policy 能不被违反的。

进一步精确：

> Policy 负责定义边界，Guard 负责在执行前用确定性规则拦住越界动作；但 Guard 只能执行写清楚的规则，不能替代策略设计、审计和人工判断。

可检查的 policy：

- 每天最多花 10 USDC。
- 只能调用白名单合约。
- 不能调用 `approve`。
- 只能在 Sepolia。
- session key 24 小时后过期。

Guard 很难直接检查的模糊 policy：

- 只做低风险操作。
- 不要乱来。
- 帮我选安全项目。
- 不要被骗。

所以 Guard 是硬边界执行器，不是万能安全保证。

小考：

1. Policy 和 Guard 的区别：
   - 我的回答：Policy 是用户定义政策的，Guard 是保证 policy 不被违反的一系列动作，前提是 policy 准确。
   - 纠偏：基本正确。更精确地说，Policy 是规则定义，Guard 是执行前的确定性检查 / 拦截层。

2. Simulation 为什么不能只说“交易会成功”：
   - 我的回答：Simulation 是假设交易进行会实际发生的细节，用于在用户 approve 前确定是否认可交易内容。
   - 确认：正确。Simulation 要告诉用户会付出什么、得到什么、哪些授权会改变、是否符合原目标。

3. Revocation 为什么不能只靠用户主动关闭：
   - 我的回答：Agent 是为了节省人工成本，对一些单一重复任务，系统可以撤销；如果所有关闭动作都人工点，浪费时间。
   - 纠偏：自动撤销不只是为了省事，更是为了及时止损。用户不一定实时看到异常，所以系统应在 session key 到期、额度用完、多次失败、异常地址、行为偏离原任务时自动收紧或撤销权限。

### 8. Human Check / Audit Log / 受限支付钱包

Handbook 这一段的核心：

- 所有自动化权限都应该有一个用户看得见的关闭入口。
- Human Check 不是所有步骤都手动确认，而是在关键风险点让用户看懂并决定。
- 超出 policy 的动作应该直接拒绝，而不是问用户“要不要继续”。
- Agent 受限支付钱包的重点是把有限授权、自动执行、随时撤销、可追踪串起来。

小考：

1. 为什么 Human Check 不是“所有步骤都让用户确认”？
   - 我的回答：低风险动作系统能直接 check 并发起，不需要用户确认；中高风险且无法直接 revoke 的动作再叫人工。这样能结合人和 AI，也降低人工成本。
   - 确认：正确。Human Check 的目标不是增加弹窗，而是在关键风险点让用户真正理解并决定。

2. 如果 Agent 超过 policy，一天最多 5 USDC 但想花 6 USDC，系统应该询问还是拒绝？
   - 我的回答：询问用户是否继续有诱导用户打破规则的隐患。应该明确拒绝，并提示如果想更改决策，要去设置里更改。
   - 确认：正确。Policy 是硬边界，不是每次执行时可以被临场绕过的建议。

3. Audit Log 有什么用？
   - 我的回答：Audit Log 是每次支付之后留下的记录，是为了追责和复盘。
   - 确认：正确。它不是为了好看，而是为了事后证明 Agent 做过什么、为什么做、有没有越界、失败在哪里。

### 9. Agent 受限支付钱包最小实践

Handbook 的最小实践场景：

> 用户允许 Agent 在一天内最多花 5 USDC，用来支付某个白名单 API 或工具服务。Agent 可以自动完成小额支付，但不能转账给任意地址，也不能修改授权。

需要展示四种 case：

| Case | Agent 想做什么 | 结果 |
|---|---|---|
| 正常 | 额度内支付给白名单 API | 执行并留下记录 |
| 超限 | 支付 6 USDC | 被 policy 拒绝 |
| 异常 | 支付给非白名单地址 | 被 Guard 拦截 |
| 撤销 | 用户撤销 session key | Agent 失去执行能力 |

小考：

1. 受限支付钱包 policy 至少包含哪三个限制？
   - 我的回答：限制金额、限制网络、限制名单策略。
   - 确认：正确。补充完整一点，还应包含动作限制（只能 `pay()`，不能 `approve` / `transfer` 任意地址）和时间限制（例如 24 小时过期）。

2. 为什么超限支付要直接拒绝，而不是让用户临时确认？
   - 沿用前一题答案：询问用户继续会诱导用户临场打破自己设置的安全边界。

3. 哪个 case 最能体现用户最终控制权？
   - 我的回答：revoke session key。
   - 确认：正确。撤销 session key 说明 Agent 的行动空间最终仍由用户控制。

### 10. Machine Payment：不只是 Agent 会转账

Handbook 核心：

> Machine Payment 讨论的是 Agent、API、服务和钱包之间如何自动完成报价、授权、付款、收据和预算控制。重点不是“AI 会花钱”，而是让机器之间的支付可限制、可验证、可追踪。

第一性原理：

> Agent 不应该拥有无限支付能力，只应该拿到具体任务、预算和收款方范围内的支付权限。

小考：

1. 为什么 Machine Payment 不是“Agent 会转账”这么简单？
   - 我的回答：如果直接让 Agent 转账，不把流程拆开和模块化判断，会有很大风险。
   - 确认：正确。Machine Payment 要处理报价、预算授权、付款、交付、验收、收据、退款 / 争议，不只是 transfer。

2. Payment Intent 和 settlement 有什么区别？
   - 我的回答：Payment Intent 是付款意图，包含预算、服务商、限制额度等；Settlement 是实际付款，包括接受 quote、支付、服务方收到 proof。
   - 确认：正确。Payment Intent 是授权上下文，Settlement 才是真正结算。

3. 合格 quote 至少包含什么？
   - 我的回答：服务内容、报价、币种等。
   - 确认：正确。补充完整：服务内容、价格、币种、收款地址、有效期、交付条件、退款条件和 quote id。

关键理解：

```text
Agent Wallet 解决：Agent 能不能花钱，怎么限制。
Machine Payment 解决：Agent 为什么花钱，付给谁，买了什么，怎么验收。
```
