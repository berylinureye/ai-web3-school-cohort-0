## 关键概念 / 学到的东西

### 1. 开发栈不是工具清单，而是工程链路

Handbook 的核心句子：

> Web3 开发栈不是一组随机工具名，而是一条从写合约、测合约、部署合约、连接钱包、调用合约到监控结果的工程链路。

对应到我已经做过的 Counter / WETH9 实操：

| 链路环节 | 我做过的动作 | 对应工具 / 概念 |
|---|---|---|
| 写合约 | 写 Counter Solidity 源码 | Remix / Solidity |
| 编译 | 生成 bytecode 和 ABI | Compiler |
| 部署 | 发起 Sepolia 部署交易 | MetaMask / RPC / Sepolia |
| 调用 | 调 `increment()` / WETH9 `deposit()` | ABI / 合约地址 |
| 验证 | 查看源码、tx、event、storage | Blockscout |

开发栈的目标不是“会用更多工具”，而是把链上不可逆执行之前的错误尽量前移到编译、测试、模拟、code review 和测试网阶段。

### 2. ABI 是链外世界和链上合约之间的翻译字典

我的初始理解：ABI 是给前端用的，让前端更好理解合约并传参。

纠偏后更准确的说法：

- ABI 不是给 EVM 用的，主要是给链外工具用的：前端、钱包、区块浏览器、脚本、AI Agent。
- 合约地址回答“我要找链上的哪一个合约账户”。
- ABI 回答“这个合约有哪些函数、每个函数要传什么参数、返回什么类型”。

如果前端拿到了正确合约地址，但 ABI 错了：

- UI 可能展示不出正确按钮或字段。
- 参数输入框和编码顺序可能错。
- 交易可能 revert，并消耗部分 gas。
- 更危险时，用户可能看不懂自己实际签的是什么。

### 3. ERC-20 token 不存在钱包里，而存在合约 storage 里

一开始我以为 ERC-20 token “存在钱包里”。纠偏后模型变成：

```solidity
mapping(address => uint256) balances;
```

ERC-20 余额本质上是代币合约 storage 里的一张余额表：

```text
0x我的地址 => 1000
0x别人地址 => 500
```

钱包不存 ERC-20。钱包只是：

- 管理私钥。
- 显示某些合约里属于我地址的余额。
- 帮我签名发交易。
- 调用 ERC-20 合约的 `transfer` / `approve` / `balanceOf`。

所以“发币”的本质不是凭空造一个东西放进钱包，而是部署一个管理余额表和转账规则的智能合约。

## 关键收获

- 钱包不存 ERC-20，钱包只是钥匙和显示器；ERC-20 余额存在代币合约的 storage 里。
- ABI 是链外工具调用合约的翻译字典；字典错了，轻则 UI 看不懂，重则用户签错交易。
- Dev Stack 的核心价值是把链上不可逆错误提前暴露，而不是收集工具名。
