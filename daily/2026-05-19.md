## 关键收获（晚上 Q&A 沉淀）

### 1. 智能合约 = 部署在链上的程序，不是"自动执行的合同"

"Smart Contract" 是营销词，**它不智能也不是合同**。

本质：跟 0518 那笔 ETH 转账走完全同一条管道（钱包签名 → RPC 广播 → 节点执行 → 改状态 → 上链 → explorer 可查），只是节点**多跑一步**：执行合约里部署好的 EVM 字节码。

跟普通后端代码的关键差异：

| 普通后端代码 | 智能合约 |
|---|---|
| 跑在你家服务器 | 跑在**全网共享的公共服务器**（任何人能看代码、能调函数）|
| 改 DB 是私有动作 | **改状态是公开历史**（不可删除、可被全网索引）|
| 跑一次几乎免费 | **全网每个节点都要算一遍**，所以要付 gas |
| 部署后随时改 | **基本改不动**（除非提前在代码里留了升级机制） |
| owner 权限随便设 | **权限必须显式写在代码里**，且公开可查 |

Handbook 那句最值得记的话：
> **"智能合约把规则变成公共基础设施，也把 bug 变成公共风险。"**

### 2. EOA / 合约账户 / EVM 三者的正确关系

之前把"EVM 是基于智能合约"理解反了。正确关系：**智能合约跑在 EVM 上，不是反过来**。

而 EOA 和 EVM 不是一个层级——EOA 是"账户类型"，EVM 是"执行环境"。正确的对比是 **EOA vs 合约账户**：

```
        ┌─────────────────────────────────────────┐
        │  EVM (Ethereum Virtual Machine)         │  ← 链上 CPU / 执行环境
        │  跑所有交易、检查签名、收 gas、改状态     │
        └─────────────────────────────────────────┘
                      ↑               ↑
              它会处理两种账户的交易：
                      │               │
        ┌─────────────┴───┐   ┌──────┴────────────┐
        │  EOA            │   │  Contract Account │
        │  公钥/私钥       │   │  没私钥           │
        │  钱包持有        │   │  有部署时写死的代码│
        │  能主动发交易    │   │  只能被调用       │
        └──────────────────┘   └────────────────────┘
```

| | **EOA**（如 MetaMask Account 1/2） | **合约账户**（如 Uniswap Router） |
|---|---|---|
| 谁控制 | 持有私钥的人 | 部署时的代码逻辑 |
| 能否主动发交易 | ✅ | ❌ 只能被调用 |
| 有没有代码 | ❌ | ✅ EVM 字节码 |
| 有没有余额 | ✅ | ✅ |
| 地址算法 | 私钥推公钥→哈希后 20 字节 | 部署者地址 + nonce 哈希 |

**关键点**：两者都是"账户"，区别在"谁决定收到调用时怎么响应"——EOA 靠钱包签名拍板，合约靠**部署时就写死的代码**自动响应。

三种典型调用：
1. **EOA → EOA 转账**（0518 做过）：`data` 为空，EVM 不跑代码，只改余额
2. **EOA → 合约调用**：`to` 是合约地址，EVM 加载字节码执行函数
3. **合约 → 合约调用**：在第 2 种交易内部发生，**不需要新签名**，但 gas 还是从最初的 EOA 扣，任意一步 revert 整笔回滚

这就是 **"可组合性"** 的物理基础——合约可以拿其他合约当乐高拼，全自动、原子性。EOA 做不到（没法在一笔交易内多步操作）。

### 3. EVM ≠ 区块链 = 链条，EVM 只是其中"执行层"

区块链至少 5 层：

| 层 | 干什么 |
|---|---|
| 网络层 (P2P) | 节点互相同步交易和区块 |
| 共识层 (PoS/PoW) | 决定谁出块、什么顺序进区块 |
| **执行层 (EVM)** | **给一笔交易，执行后状态变什么** |
| 数据层 | 区块/状态/历史数据存哪儿 |
| 账户/状态存储 (Merkle Patricia Trie) | 全网所有账户状态的高效哈希结构 |

EVM 只管执行层。共识、出块、链怎么连——它都不管。

**反例佐证**：
- Bitcoin 是区块链，但**没有 EVM**（用 Bitcoin Script，几乎不能跑应用）
- Solana 是区块链，但用 **SVM**（Solana VM）不用 EVM
- Arbitrum / Base / Optimism 是**EVM 兼容链**——共识层是 rollup，执行层都跑 EVM，所以以太坊合约能直接搬过去

所以更准确的话：**"EVM 是以太坊系链共享的'执行核心'"**。"链条"指的是数据结构（区块按哈希指针链起来），跟 EVM 是两件事。

### 4. 合约账户的优势：分两种角色看

直觉问的"合约账户感觉没私钥安全"——其实把两个不同对象比错了。要拆开看：

**角色 A：合约账户 = 链上应用本身（绝大多数情况）**
Uniswap / Aave / ERC20 / NFT / DAO 都是合约账户。**它们不是某个人的钱包，是应用**。
EOA 根本干不了这件事——EOA 只能签名转账，没法跑撮合/借贷/投票逻辑。所以这条线上**没有"EOA vs 合约谁安全"的比较**，合约的"优势"等于"它的存在让链上应用层可能"。

**角色 B：合约账户 = 智能钱包（账户抽象 / ERC-4337）**
这才是直觉真正问的。答案：**EOA 适合简单场景，合约钱包适合复杂场景**。

| | EOA | 合约钱包（Safe / ERC-4337） |
|---|---|---|
| 安全模型 | **all-or-nothing**：私钥在=钱在，丢=全丢 | **可编程**：规则写代码里 |
| 私钥泄漏 | 资产全失 | 多签可救（攻击者拿 1/3 不够）|
| 助记词丢 | 永久失去账户 | **社交恢复**：3 个朋友确认换控制权 |
| 防大额误操作 | 没有 | 合约可写"日限额 100 USDC，超额延迟 24h" |
| 给别人/AI 临时授权 | 没有（只能整把私钥给）| **session key**：限时间/金额/白名单 |
| 紧急暂停 | 没有 | 合约可设 emergency pause |
| Gas 怎么付 | 必须 ETH | 可让别人代付（gasless），可用 USDC 付 |
| **谁更安全** | **简单/冷钱包/低金额** | **团队金库 / DAO / AI Agent 代管 / 大额长期持有** |

→ **核心洞察**：EOA 是 2015 年硬编码的账户类型，规则只有一条——"谁有私钥谁说了算"。**账户抽象（Account Abstraction）就是把"账户的规则本身变成可编程的"**。

**对产品研究方向特别重要**：
> **AI Agent 代管资产，几乎只能用合约账户**。把 EOA 私钥交给 Agent = Agent 跑路血本无归；把合约账户的 session key 交给 Agent = Agent 最多在你定的边界里乱来，超出边界合约自己 revert。

这直通长期方向 [bridge/agent-wallet](https://aiweb3.school/zh/handbook/bridge/agent-wallet/)。MetaMask 是 EOA，所以现在的实操经验在 Day 6 学到 agent-wallet 时需要做一个迁移——换成合约钱包（如 Safe）。

## 今晚追加学习：Handbook smart-contract 章节 5 段精读

> 5 个 task 提交完后，自己感觉概念还没真掌握，又跟 Agent 走了一轮"读 → 用自己的话总结 → Agent 纠偏"的主动学习循环。把今晚误打误撞的 Q&A 模式换成 cohort 训练营推荐的学法。

### A. ABI（Application Binary Interface）

**自己的总结**：是给"用户"用的，告诉能调什么、没告诉是否安全；和 skill 比差在不是自然语言。

**Agent 纠偏**：
- "给用户用的" 不准 → 准确是**给软件工具用的**（前端 SDK、区块浏览器、AI Agent），人类通过工具间接消费
- "能调什么、不告诉是否安全" ✓ Handbook 原话："**ABI 是机器可读接口，不是安全说明书**"
- "和 skill 差在格式" 是表面差，**真正核心差是**：skill metadata 含语义指引（when to use / 风险提示 / 业务前提），ABI 只有结构（函数名 + 参数类型），完全没语义
- **产品研究 takeaway**：AI Agent 拿到 ABI ≈ 拿到一个只有函数签名、没有 docstring 的 SDK——能调，但不知道**该不该调、什么时候调、调了会怎样**。要让 Agent 安全自主调合约，必须在 ABI 之上额外喂语义层（合约文档、安全提示、白名单）。这是 bridge/web3-tool-use 章节话题。

### B. Event

**自己的总结**：event 类似落下的日志，记录每一个行为，落在数据库，可追溯；storage 类似于存储了。

**Agent 纠偏**：
- "落在数据库" 不准 → event 落在**区块的 receipts logs 区**（每笔交易的回执里）；前端能秒查的"数据库"是**链下索引器**（如 The Graph）从链上抓 events 重组的
- "storage 类似存储" 太模糊 → 真正的区别：

| 维度 | Storage | Event |
|---|---|---|
| 是什么 | **当前状态快照**（"USDC 谁有多少"） | **历史动作日志**（"USDC 每次转账"） |
| 合约自己能读吗 | ✓（合约下次执行能拿值） | ✗（emit 完就送出去了） |
| Gas 成本 | 超贵（改 1 slot 几千-2 万 gas） | 便宜（emit 1 条几百-1500 gas） |

- **关键 invariant**：合约自己读不到 event。**如果合约后续逻辑要"知道某件事发生过"，必须存 storage**，不能靠 event
- **产品研究 takeaway**：Uniswap / OpenSea / Etherscan 前端 99% 数据查询走 events 重建的链下 DB（直接扫 storage 又慢又贵）。**合约 event 设计质量直接决定 dApp 能不能做出快速 UI**。看一个项目"前端工程师好不好招"，可以反过来看 event 设计干不干净

### C. Upgrade

**Trade-off**：
- 不可升级：规则真正固定 (代码即法律)，但 **bug = 资产永久损失**（The DAO 2016 真实案例，导致 ETH/ETC 硬分叉）
- 可升级：可迭代修 bug 加功能，但**背后仍有"人"能改规则**，本质是"信任管理员/治理"

**Handbook 给的 4 个判断问题对应防什么风险**：

| 问题 | 防的风险 |
|---|---|
| 升级权限在谁手里？ | 防"单个 EOA 跑路 = 全协议归零" |
| 用户能看到升级提案和新实现地址吗？ | 防"管理员深夜偷偷上线新版掏空"（rug pull） |
| 升级能改资产转移/提款/暂停/权限吗？ | 防"管理员重写规则锁死用户" |
| 管理员密钥泄漏最坏结果是什么？ | 评估需要给这个项目多少信任溢价 |

**4 个问题串起来 = 一张"信任尽调表"**——看一个 dApp 实际去中心化程度的 5 分钟体检。

### D. 一个调用流程怎么发生（8 步链路）

把今晚所有概念串成一张图：

```
用户点按钮 → [1] 前端读地址+网络 → [2] 前端用 ABI 编码 → calldata
        → [3] 钱包弹窗签名 → [4] 广播 RPC → [5] 验证者打包
        → [6] EVM 执行（改 storage + emit event；revert 则状态回滚但 gas 不退）
        → [7] 前端监听 tx receipt → [8] 索引器读 event → 链下 DB → 前端查询
```

**测验**：第 6 步 revert，会扣 gas 吗？答：**会**，因为 EVM 已经付出了计算成本。
- 实际消耗到 revert 那一刻的 gas 不退；预留但未用的 gas 会退
- 设计原因：**反 DoS**。如果失败交易不花钱，攻击者可零成本灌满区块空间
- **takeaway**：Web3 安全 ≠ 只看代码，**经济激励层也是安全设计的一部分**

### E. 常见错误（5 个，按产品研究方向相关度排）

| Handbook 错误 | 产品研究方向相关度 | 原因 |
|---|---|---|
| **#4 把 owner 当成永远可信** | ⭐⭐⭐⭐⭐ | **直接可评估**（读合约 / 用 DeFiSafety 等工具），且跟 Upgrade 4 问同源；所有 rug pull 根源 |
| #5 忽略事件设计 | ⭐⭐⭐⭐ | 间接但能判断"前端能做到多好"（dApp 体验信号） |
| Upgrade 4 问 | ⭐⭐⭐⭐ | 跟 #4 同源，决定"实际去中心化程度" |

### F. 广播（Broadcast）—— 之前一直没单独讲清

**定义**：把已签名的交易扔给某个 RPC 节点，让它通过 P2P gossip 协议在区块链网络里扩散传播，最终被验证者看到并打包进区块。**没有广播这一步，签好的交易只是抽屉里一张纸**。

**子步骤**：钱包 → (1) JSON-RPC HTTP → RPC 节点 → (2) devp2p gossip → 其他节点 → (3) mempool → (4) 验证者选 → (5) 打包上链。第 (1) 步你能控制，后面不能。

**三个深层意义**：
1. **RPC 节点能"看见你"**：MetaMask 默认走 Infura，**一家公司能看到全网 ~50% 钱包交易流**。"去中心化"在协议层；**接入层（RPC）依然由少数公司掌控**
2. **Mempool 公开 → MEV**：广播后交易在 mempool 公开，"搜索者"程序 7x24 监控，付更高 gas 抢先上链（**front-running**）。整个 MEV 行业年规模 10 亿美元级
3. **私有广播是新基础设施**：Flashbots Protect / MEV-Share / SUAVE 把交易直接发给区块构建者，绕过公开 mempool，避免被 MEV bots 抽水

**对 AI Agent 方向**：Agent 自动化交易时，"用哪个 RPC 广播"是关键决策——公开 RPC 暴露 MEV，私有 RPC 保护但复杂，自建节点最强但维护贵。Agent 越主动、自动化越深，被 MEV 抽水次数越多。这是 bridge/agent-workflow 章节设计点。
