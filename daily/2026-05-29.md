### 1. 待补：Indexing

Handbook 核心：

> 链上数据是公开的，但不等于好用。Indexing 的作用，是把区块、交易、事件和合约状态整理成产品、分析工具和 AI Agent 能快速查询的结构化数据。

第一性原理：

> 产品需要的是面向问题的数据模型，而不是原始区块流。

小考：

1. 为什么“链上数据公开”不等于“产品可以直接好用”？
   - 我的回答：链上数据可以理解为某个时间点、局限信息的快照；用户层 / 产品层更关注一段时间内交易信息的汇总和可视化，这些不是链上数据可以直接触达的。
   - 纠偏：方向正确。更精确地说，链上不是静态快照，而是原始区块 / 交易 / logs 流。产品需要的是按问题组织后的历史视图，例如地址交易历史、协议 TVL、Agent 执行记录、订单状态。

2. Event Indexing 是什么？
   - 我的回答：Event indexing 类似于把数据结构化，增加一些面向用户关注点的查询能力；如果 Web3 结合 AI Agent，event indexing 可以给 Agent 更好的上下文。
   - 确认：正确。Event 是合约对外发出的业务日志，索引器把 `Transfer`、`Deposit`、`PaymentReceived` 等 event 转成数据库表 / 查询接口。

3. 为什么 AI Agent 不能只拿一个没有来源和时间戳的链上数据结论？
   - 我的回答：因为来源可靠性很重要。
   - 确认：正确。补充：Agent 需要知道数据来自哪个合约 / tx / block / indexer，更新时间是什么，否则可能基于过期或错误数据推理。

关键理解：

```text
链上 = 事实来源
索引层 = 可用数据层
Event = 合约给索引器的业务信号
RPC = 读链和发交易接口，不是完整数据库
AI Agent = 需要带来源、时间戳、可回溯的结构化链上上下文
```

### 2. Web3 Security：不是上线前审计一次

Handbook 核心：

> Web3 安全不是上线前找人审一次代码，而是从合约设计、权限、测试、交易模拟、监控、应急暂停到权限撤销的一整套工程流程。

第一性原理：

> 链上系统默认暴露在公开对抗环境里，任何可调用路径都要按攻击面看待。

小考：

1. 为什么 Web3 安全不是“上线前审计一次”就够了？
   - 我的回答：上线前审计只适用于 Web2 时代；Web3 所有 public 函数都向攻击者公开，一旦漏洞没及时修复，资金会不安全。
   - 纠偏：审计不是 Web2 才有，Web3 也非常需要审计。但审计只是安全流程的一环，不是安全保证书。Web3 还需要权限设计、测试、simulation、monitoring、pause、revocation 和上线后响应。

2. “任何可调用路径都要按攻击面看待”是什么意思？
   - 我的回答：因为 Web3 所有对外开放的调用路径都有可能被攻击者调用，因此要换位去看待。
   - 确认：正确。不能因为前端没展示按钮，就假设函数不会被调用；只要链上可调用，攻击者就能直接调合约。

3. 如果 Agent 生成了一笔交易，执行前至少应该做哪三类安全检查？
   - 我的回答：policy 权限检查、simulation、human check。
   - 确认：正确。补充：还应检查网络 / 合约地址 / ABI、token 授权、金额、滑点、预言机价格和是否触发敏感权限。

关键理解：

```text
Web3 安全 = 设计前置 + 执行前模拟 + 执行中确认 + 上线后监控 + 出事后止损
Agent 生成交易 ≠ 可以执行交易
public 函数不可怕，可怕的是把限制只写在前端，而没有写进合约 / policy / guard
```

补充理解：

- 如果一个函数本来设计上允许所有人调用，并且不管谁调用都不会破坏规则，那么它公开没有问题。
- 真正危险的是开发者以为“前端不展示按钮 = 用户不能调用”，但攻击者可以直接绕过前端调用合约。
- 所以 Web3 安全要假设攻击者知道所有函数、参数和调用路径；安全边界必须在合约规则、权限检查、policy、guard 和钱包确认里。
- 对 Smart Wallet / Account Abstraction 来说，确认的重点不是“所有动作都人工点一下”，而是确认 policy 边界是否准确、simulation 是否可信、这次操作是否越权，以及高风险动作是否需要人工审核。

### 2.5 dApp：不是没有前端，而是核心规则在链上

补充理解：

```text
普通 App：核心规则和数据主要在公司服务器里
dApp：核心资产和关键规则主要在链上合约里
```

dApp 通常由三层组成：

```text
前端页面：用户点击按钮的界面
钱包：用户签名确认
智能合约：真正执行资产和规则的链上程序
```

例子：

- Uniswap 是 dApp：前端选择兑换，钱包确认签名，合约池子执行兑换。
- Aave 是 dApp：前端选择存款 / 借款，钱包确认交易，合约记录抵押、借款和清算规则。

关键区别：

```text
普通 App 的可信核心在公司后端；
dApp 的可信核心在链上合约。
```

但 dApp 也可能有中心化部分，例如前端网站、索引服务、RPC、风控系统和客服。重点不是“完全没有中心化”，而是核心资产规则是否由链上合约承载。

### 2.6 Account Abstraction：ERC-4337 流程慢拆

Handbook 核心：

> 在 ERC-4337 里，用户不直接发送普通交易，而是创建 `UserOperation`。Bundler 收集这些操作，提交到链上的 EntryPoint 合约。EntryPoint 再调用智能账户验证和执行逻辑。

最小理解：

```text
UserOperation = 操作申请表
Bundler = 把申请表送上链的人
EntryPoint = 链上的统一柜台 / 中转入口
Smart Account = 账户门禁和执行规则
Paymaster = gas 付款规则
```

小考：

为什么 UserOperation 不能自己直接变成链上交易，需要 Bundler 把它提交到 EntryPoint？

- 我的回答：因为 EntryPoint 类似于分流的柜台和中转站，很重要。
- 纠偏：EntryPoint 的确是统一入口，但 `UserOperation` 自己不会自动进入链上交易池。Bundler 才是把多个 UserOperation 收集起来、模拟检查、包装成真正链上交易并提交到 EntryPoint 的角色。

补充小考：

Smart Account 主要负责什么？

- 我的回答：没有 Bundler 打包到 EntryPoint，UserOperation 根本到不了链上，触达不了 wallet；Smart Account 负责审核这个交易是否安全，是否可以进行。
- 纠偏：方向正确。更精确地说，Smart Account 负责按账户规则验证这次操作是否有效，例如签名、nonce、权限、额度、session key、目标合约和函数限制；验证通过后再执行目标动作。它不是泛泛地“判断安全”，而是按可验证规则做门禁。

### 2.7 Paymaster：带风控的 gas 代付系统

Handbook 核心：

> Paymaster 允许第三方为用户操作支付 gas，或者让用户用非原生资产承担费用。但 Paymaster 不是免费午餐。它需要风控。

最小理解：

```text
Paymaster 不是免费 gas；
Paymaster 是带风控的 gas 代付系统。
```

它解决的体验问题：

```text
用户想用 dApp，但钱包里没有链的原生 gas token。
如果为了完成一次交互，还要先去买 ETH / MATIC / 其他 gas token，体验会很复杂。
```

小考：

1. Paymaster 解决的主要体验问题是什么？
   - 我的回答：解决的是用户进行某些币的交易时必须用链的原生币付 gas；如果用户为了完成交易还要先买原生 gas token，会非常复杂。
   - 确认：正确。Paymaster 让应用可以在规则内帮用户处理 gas，降低新用户进入 dApp 的门槛。

2. 为什么 Paymaster 不能无条件给所有人付 gas？
   - 我的回答：因为 Paymaster 需要风控，需要对交易进行审核，避免不必要损失；如果每个人都能薅羊毛，会形成很大亏损。
   - 确认：正确。Paymaster 必须限制用户、合约、函数、额度、频率和失败交易成本，否则会被 spam、套利或恶意刷补贴。

应用小考：

假设 AI Agent 每天自动执行一次小额测试网任务，Paymaster 可以帮它付 gas，应该设计哪些风控规则？

- 我的回答：先看 Smart Account 给出的结论，看交易是否可以进行；检查短时间发起交易频率，如果连续多次可能是在薅羊毛；检查要覆盖的智能合约；检查 token 是否在支持范围内。
- 确认：正确。可整理为：
  - 只赞助通过 Smart Account / policy 验证的操作。
  - 限制交易频率，短时间多次请求要限流或拒绝。
  - 只赞助白名单合约。
  - 只支持指定链、token 和业务动作。
  - 补充：设置单次 / 每日预算上限，连续失败后熔断。

### 2.8 ERC-4337 总流程

```text
用户 / Agent
  ↓ 创建
UserOperation（操作申请表）
  ↓ 由 Bundler 收集、模拟、包装成链上交易
Bundler
  ↓ 提交
EntryPoint（统一入口）
  ↓ 调用验证
Smart Account（账户规则：签名、nonce、session key、权限、额度）
  ↓ 可选 gas 验证
Paymaster（gas 赞助规则：用户、合约、函数、频率、预算）
  ↓ 验证通过
执行目标合约动作
```

最终理解：

```text
Account Abstraction 让账户从“私钥直接控制”变成“规则控制”。
ERC-4337 用 UserOperation / Bundler / EntryPoint / Smart Account / Paymaster 把这套规则流程标准化。
```

复述小考：

ERC-4337 的完整流程是什么？

- 我的回答：ERC-4337 取代用户直接通过 EOA 私钥发交易的高风险行为。用户先提出 UserOperation，类似 prompt / 需求；Bundler 把需求打包发送到 EntryPoint；EntryPoint 连接用户的 Smart Account；Smart Account 里的 AI Agent 判断交易是否可执行和风险；Paymaster 再做自己的风控，判断是否帮用户垫付 gas。它的优势包括安全把关，以及让用户不用为了某个交易专门购买原生 gas token。
- 纠偏：
  - ERC-4337 不是简单“取代 EOA”，而是提供一种智能账户交易流程，让账户验证和 gas 支付变得可编程。EOA 仍然存在，也可能作为 Smart Account 的 owner / signer。
  - UserOperation 不是完整交易，也不是私钥交易的替代品，而是标准化的“操作请求数据包”。
  - Smart Account 里不一定有 AI Agent。它主要是合约账户，按规则验证签名、nonce、session key、权限、额度、目标合约和函数。AI Agent 可以在账户外部生成请求或辅助判断，但不能默认等同于 Smart Account 的内部组件。
  - Paymaster 的确可以降低 gas token 门槛，但它不是“加速交易”的核心角色，而是“按风控规则赞助 / 处理 gas”的角色。

更准确版本：

```text
用户 / Agent 创建 UserOperation；
Bundler 收集并把它包装成链上交易，提交到 EntryPoint；
EntryPoint 按 ERC-4337 标准流程调用 Smart Account 验证；
Smart Account 按账户规则判断这次操作是否有效；
Paymaster 可选地按风控规则判断是否赞助 gas；
全部验证通过后，目标合约动作才被执行。
```

### 3. 今日产出：Week 2 Threat Model 草稿

新增：

- [`tasks/week2-agent-workflow-threat-model.md`](../tasks/week2-agent-workflow-threat-model.md)

内容覆盖：

- 资产风险：资金转走、无限授权、错误 token、滑点过大。
- 权限风险：私钥泄漏、session key 过宽、owner/admin 操作、policy 被绕过。
- 数据风险：过期索引、假数据源、oracle 延迟、token metadata 污染。
- 工具调用风险：tool abuse、高权限工具混用、浏览器诱导、文件系统越权。
- 外部依赖风险：RPC 不一致、indexer 延迟、quote 伪造、预言机异常。

确认策略：

```text
低风险只读：可自动执行并记录来源
交易草稿：可自动生成，不自动签名
小额策略内 payment：simulation 后可自动或轻确认
approve / swap / 借贷 / 跨链：必须 human check
owner / upgrade / withdraw / setOracle：必须多签 / timelock / 强确认
secret：Agent 不读取、不输出、不提交
```

Prompt injection 练习：

> 外部合约文档要求 “ignore all previous security rules” 并诱导 approve unlimited USDC。系统应把它标记为 untrusted context，拒绝生成 approve 交易，并记录 audit log / alert。

## 关键收获

- 链上数据公开不等于产品可用，索引层负责把原始区块 / logs 转成面向问题的数据模型。
- Agent 拿链上上下文时，必须附带来源、时间戳、区块高度和可回溯证据。
- Web3 安全不是审计一次，而是设计、权限、测试、simulation、monitoring、pause、revocation 的连续流程。
- Agent workflow threat model 要覆盖资产、权限、数据、工具调用、外部依赖和失败后果。
