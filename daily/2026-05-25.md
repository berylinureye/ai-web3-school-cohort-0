## 关键概念 / 学到的东西

### 1. Remix / Hardhat / Foundry / OpenZeppelin 的位置

Handbook 这一段先把工具按开发阶段分开：

- **Remix**：浏览器里的 Solidity IDE，适合入门、教学、原型和快速验证。
- **Hardhat**：适合 JavaScript / TypeScript 项目，把合约开发接进测试、脚本和前端工程。
- **Foundry**：更偏命令行和 Solidity-native 测试，适合高强度合约开发和快速反馈。
- **OpenZeppelin**：提供 ERC-20、ERC-721、AccessControl、Ownable、Pausable 等常用合约标准和安全组件。

我的理解：

- Remix 是“智能合约在线实验台”，可以在浏览器里写 Solidity、编译、部署、调用合约。
- Solidity 是写以太坊智能合约的编程语言。人写 Solidity，EVM 跑的是编译后的 bytecode。
- Git 是代码 / 笔记的版本管理工具，像“时间机器 + 协作账本”。
- CI 是 Continuous Integration（持续集成），每次 push 后自动跑编译、测试、build 等检查。
- 真实项目不能长期只靠 Remix，因为浏览器里点来点去很难形成 Git 记录、自动化测试、部署脚本和 CI。

一句话总结：

> Remix 帮我理解合约，Hardhat 帮我工程化合约，Foundry 帮我深入测试合约，OpenZeppelin 帮我少写危险的基础代码。

### 2. Git / CI / Solidity / Remix 的补充理解

今天卡住的几个基础词：

| 词 | 一句话解释 |
|---|---|
| Solidity | 写以太坊智能合约的编程语言 |
| Remix | 浏览器里的 Solidity 合约开发实验台 |
| Git | 记录代码和笔记怎么变化的版本管理工具 |
| CI | 自动检查代码有没有坏掉的持续集成流程 |

Solidity 到链上的流程：

```text
Solidity 源码
   ↓ 编译
Bytecode + ABI
   ↓ 部署
EVM 执行 bytecode
```

Remix 把这条流程用浏览器 UI 包起来，让初学者不用先搭完整工程，也能先看懂“合约从代码到上链”的全过程。

### 3. ABI vs viem：说明书和执行工具

今天问到：viem 和 ABI 有什么区别？

更准确的模型：

- **ABI**：合约的“说明书 / 翻译字典”。它描述合约有哪些函数、参数类型、返回值类型。
- **viem**：拿着 ABI、合约地址和 RPC 去真正读链 / 写链的 TypeScript 工具库。

类比：

```text
合约地址 = 餐厅地址
ABI = 菜单，告诉你能点什么菜、每道菜要什么参数
viem = 服务员 / 点餐系统，拿着菜单帮你下单、查订单状态
钱包 = 付款和签字的人
RPC = 把请求送到链上的通道
链 = 真正执行规则的厨房
```

对非程序员 / 产品研究视角来说，dev-stack 不需要按“我会不会写代码”理解，而要按“风险地图”理解：

```text
Solidity / 合约：规则有没有写错？
ABI：前端理解合约有没有错？
合约地址：用户是不是在跟正确合约交互？
viem / wagmi：前端调用和状态管理有没有错？
钱包：用户签名前有没有看懂？
RPC：请求是不是发到了正确网络？
Blockscout：事后能不能验证？
Git / CI / 测试：上线前有没有把错误拦住？
```

一句话：

> 我不用成为 viem 工程师，但要能判断一个 dApp 或 Agent Wallet 在“读、签、发、确认、验证”这几步里有没有把用户风险讲清楚、控制住。

### 4. viem vs wagmi：链上动作能力和前端状态管理

小考纠偏：

- 我答对了：如果用户分不清“等待签名”和“等待上链”，主要是 wagmi / 前端状态设计问题。
- 我答对了：必须先签名，交易才可能广播并进入等待确认阶段。
- 需要纠偏：viem 不是“写合约的工具”，也不是帮助合约写得更好。写合约是 Solidity / Remix / Hardhat / Foundry 那一侧。viem 是前端或脚本拿着 ABI 和合约地址去读链 / 写链的工具。

更准确的区分：

| 工具 | 它站在哪一侧 | 解决什么问题 |
|---|---|---|
| ABI | 合约接口说明 | 告诉链外工具合约有什么函数、参数和返回值 |
| viem | 前端 / 脚本与链交互 | 读合约、写合约、编码调用、查交易状态 |
| wagmi | React dApp 前端状态 | 管理钱包连接、当前账户、当前网络、交易状态和 UI hooks |

一句话：

> viem 是“怎么真的去调链”，wagmi 是“页面怎么管理用户正在经历的链上流程”。

复考题：

> 如果一个页面要显示“我的 ERC-20 余额”，它需要合约地址 + ABI + 用户钱包地址 + 什么？

我的回答：**viem**。

纠偏后确认：答对。读取 ERC-20 余额的核心动作是“拿着合约地址、ABI 和用户地址去调用 `balanceOf(address)`”，这更接近 viem 的职责。如果这个读取发生在 React 页面里，wagmi 可能会把 viem 的底层能力包装成 loading / success / error 等前端状态。

### 5. 一条具体工具链怎么串起来

Handbook 用最小投票合约举例：

1. 用 Remix 写第一版 `Voting.sol`，确认语法和基本调用流程。
2. 把合约迁移到 Hardhat 或 Foundry repo。
3. 写测试覆盖创建投票、重复投票、投票结束、非管理员操作失败。
4. 用本地链部署，记录地址和 ABI。
5. 用 viem 或 wagmi 写前端，读取候选项、发起投票交易、展示 pending 和 confirmed 状态。
6. 部署到测试网后，在区块浏览器验证源码，并检查 event 是否能被正确查询。

理解检查：

- 不能直接“写页面调合约”，因为这样会隐藏很多问题。Remix 帮我先看清合约逻辑，测试帮我确认权限和失败路径，本地链帮我在可控环境里复现部署和调用。
- “重复投票”和“非管理员结束投票”不是语法问题，而是**权限 / 业务逻辑问题**。
- Blockscout 在最后帮我验证：源码、交易、event。

小纠偏：

> 这里应说 ABI，不是 API。API 是一般的软件接口概念；ABI 是智能合约给链外工具使用的函数 / 参数 / 返回值说明。

### 6. AI x Web3 中的位置：能力越强，边界越要明确

Handbook 这一段的核心：

> AI 可以显著提升 Web3 开发效率，但 AI 参与开发栈后，验证流程反而要更明确。

我的理解检查：

- AI 可以帮我解释 ABI。
- AI 可以帮我写测试。
- AI 可以帮我写合约草稿和本地部署流程。
- 但 AI 可能静默做错事，所以执行边界必须更明确。
- 签名、私钥、链上交易确认、授权等动作必须人工确认。

补充边界：

- **可以让 AI 做**：读 Handbook、解释合约、生成测试、写部署脚本、排查交易失败、总结 owner / mint / pause 等权限。
- **需要人工确认**：任何钱包签名、任何会消耗 gas 的交易、任何 approve / permit / setApprovalForAll、任何测试网或主网部署。
- **不能交给 AI 静默处理**：私钥、助记词、API key、`.env`、owner 权限转移。

一句话：

> AI 在 Web3 里不是“越自动越好”，而是“在明确边界里自动，在资产动作前停下来让人确认”。

## 关键收获

- Solidity 是“写链上规则”的语言；链上真正跑的是它编译出来的 bytecode。
- Remix 是智能合约的在线实验台，适合建立手感，但不适合长期替代工程化 repo。
- Git 记录变化，CI 自动检查变化有没有把项目弄坏。
- ABI 是说明书，viem 是拿着说明书去读链 / 写链的工具。
- 对我来说，dev-stack 更像 dApp / Agent Wallet 的风险地图，而不是工程工具清单。
- viem 不是写合约工具；它是前端 / 脚本调用合约的工具。
- wagmi 更靠近用户体验，负责让页面清楚区分连接钱包、签名、广播、确认、成功和失败。
